# Evento — Event Sourcing & CQRS for Rust

Evento is an event-sourcing toolkit built around an `Executor` abstraction (SQLite / MySQL / PostgreSQL via sqlx, or Fjall embedded). Events are serialized with [`bitcode`](https://crates.io/crates/bitcode); aggregator/handler boilerplate is generated by `evento-macro`.

Source: <https://github.com/timayz/evento>

Companion references in this directory:

- [`macros.md`](./macros.md) — `#[evento::aggregator]`, `#[evento::projection]`, `#[evento::handler]`, `#[evento::subscription]`, `#[evento::subscription_all]`, `#[derive(Cursor)]`: what they expand to.
- [`subscriptions.md`](./subscriptions.md) — `SubscriptionBuilder`, routing keys, retries, shutdown, `Context`, `Data` injection.
- [`executors.md`](./executors.md) — SQL pool setup, migrations, Fjall, `Evento` / `EventoGroup` / `Rw`, raw reads with `Reader`.

## Cargo setup

```toml
[dependencies]
evento  = { version = "2", features = ["sqlite"] }   # or "postgres", "mysql", "fjall"
bitcode = "0.6"
sqlx    = { version = "0.8", features = ["runtime-tokio", "sqlite"] }
tokio   = { version = "1", features = ["full"] }
anyhow  = "1"
```

Feature flags on the `evento` crate:

| Feature   | Effect |
|-----------|--------|
| `macro`   | Procedural macros (on by default) |
| `sql`     | Enables `sqlite` + `mysql` + `postgres` |
| `sqlite`  | `sqlx::Sqlite` executor + migrator |
| `mysql`   | `sqlx::MySql` executor + migrator |
| `postgres`| `sqlx::Postgres` executor + migrator |
| `fjall`   | Embedded `Fjall` executor (no migrator) |
| `group`   | `EventoGroup` multi-executor reads |
| `rw`      | `Rw<R, W>` read/write-split executor |

## Mental model

```
                  ┌────────────────┐
write API ──────► │   Executor     │ ◄────── read API (Projection / SubscriptionBuilder)
(create/aggreg.)  │ (SQL / Fjall)  │
                  └────────────────┘
                           ▲
                           │  events keyed by aggregator_type + aggregator_id + version
                           │  optimistic concurrency on `original_version`
```

- **Aggregator** — a type identifier (`"my_crate/BankAccount"`). Each aggregate *instance* has an ID (typically a ULID).
- **Event** — an immutable fact persisted under one aggregate at an incrementing `version`. Serialized as bitcode.
- **Projection** — a read model rebuilt by replaying events through `#[evento::handler]` functions.
- **Subscription** — a long-running task that streams new events through handlers and acknowledges its cursor.
- **Executor** — the storage backend. All write/read APIs are generic over `Executor`.

## Defining events — `#[evento::aggregator]`

Each variant becomes its own public struct. The enum becomes a unit struct used as the aggregator handle.

```rust
#[evento::aggregator]
pub enum BankAccount {
    /// doc comments are preserved on the generated struct
    AccountOpened { owner_id: String, initial_balance: i64 },
    MoneyDeposited { amount: i64, transaction_id: String },
    MoneyWithdrawn { amount: i64 },
    AccountClosed,                       // unit variant → unit struct
}
```

Expansion (sketch):

- `pub struct AccountOpened { pub owner_id: String, pub initial_balance: i64 }` (+ for each variant)
- Derives auto-added: `Debug, Clone, PartialEq, Default, bitcode::Encode, bitcode::Decode`
- `impl evento::Aggregator for AccountOpened` returning `aggregator_type() = "{cargo_pkg_name}/BankAccount"`
- `impl evento::AggregatorEvent for AccountOpened` returning `event_name() = "AccountOpened"`
- `pub struct BankAccount;` with `impl evento::Aggregator` (used in `Projection::new::<BankAccount>(id)`)

**Aggregator type is `"{CARGO_PKG_NAME}/{EnumName}"`** — derived at compile time. Renaming the crate changes the persisted type string, so plan for that.

Pass extra derives with `#[evento::aggregator(serde::Serialize, serde::Deserialize)]`.

## Writing events — `create()` / `aggregator()`

`evento::create()` builds events for a brand-new aggregate (ULID auto-generated). `evento::aggregator(id)` targets an existing aggregate and **requires `original_version`** for optimistic concurrency.

```rust
use evento::metadata::Metadata;

// New aggregate
let account_id = evento::create()
    .event(&AccountOpened { owner_id: "u1".into(), initial_balance: 1000 })
    .metadata(&Metadata::default())
    .routing_key("accounts")          // optional; locked once set
    .commit(&executor)
    .await?;

// Append to an existing aggregate
evento::aggregator(&account_id)
    .original_version(1)              // current version BEFORE this commit
    .event(&MoneyDeposited { amount: 100, transaction_id: tx.into() })
    .event(&MoneyDeposited { amount: 50,  transaction_id: tx2.into() }) // multi-event batch is atomic
    .commit(&executor)
    .await?;
```

Builder methods:

| Method | Purpose |
|--------|---------|
| `.event(&E)` | Append an event (must impl `AggregatorEvent + bitcode::Encode`); multiple calls = atomic batch |
| `.original_version(u16)` | Expected current version; mismatch → `WriteError::InvalidOriginalVersion` |
| `.routing_key(impl Into<String>)` / `.routing_key_opt(Option<String>)` | Partition key; locked after first set |
| `.metadata(key, &value)` | Insert a bitcode-encoded metadata key |
| `.metadata_from(impl Into<Metadata>)` | Replace the whole metadata blob |
| `.requested_by(user_id)` / `.requested_as(user_id)` | Standard metadata setters |
| `.commit(&executor)` | Persist; returns the aggregate ID |

`AggregatorBuilder::ids(vec![...])` derives the aggregate id by SHA3-hashing a list of input IDs (useful for relationship aggregates).

`WriteError` variants: `InvalidOriginalVersion`, `MissingData` (no `.event()` called), `Unknown(anyhow::Error)`, `SystemTime(...)`.

**Routing key is "sticky" on the aggregate.** The builder reads the *first* event of the aggregate and reuses its `routing_key`; the `.routing_key(...)` call on appends is ignored if events already exist.

## Reading state — `Projection`

A projection rebuilds a read model by replaying events.

```rust
use evento::{metadata::Event, projection::Projection};

#[evento::projection]                    // adds `pub cursor: String` + ProjectionCursor impl
#[derive(Debug)]
pub struct AccountView {
    pub balance: i64,
    pub owner: String,
}

#[evento::handler]
async fn on_account_opened(
    event: Event<AccountOpened>,
    view: &mut AccountView,
) -> anyhow::Result<()> {
    view.owner = event.data.owner_id.clone();
    view.balance = event.data.initial_balance;
    Ok(())
}

#[evento::handler]
async fn on_money_deposited(
    event: Event<MoneyDeposited>,
    view: &mut AccountView,
) -> anyhow::Result<()> {
    view.balance += event.data.amount;
    Ok(())
}

let account: Option<AccountView> =
    Projection::<_, AccountView>::new::<BankAccount>(&account_id)
        .handler(on_account_opened())
        .handler(on_money_deposited())
        .safety_check()                  // fail if an unhandled event is seen
        .execute(&executor)
        .await?;
```

Notes:

- `Projection::new::<A>(id)` — first generic is `Executor` (often inferred), second is the projection state type; `A` is the aggregator handle (the unit struct from `#[evento::aggregator]`).
- `.handler(handler_fn())` — call site is `handler_fn()` (the macro generates a zero-sized constructor of the same name).
- `.safety_check()` — by default unknown events are skipped. With safety on, an unhandled event makes `execute` return `Err`.
- `.aggregator::<Owner>(owner_id)` / `.aggregator_raw(type, id)` — pull events from additional aggregates into the same projection.
- `.data(value)` — insert shared data accessible via `Context::extract` inside handlers.
- `.skip::<MyEvent>()` — register a no-op handler so safety_check passes for events you don't need.
- `.revision(u16)` — bump to invalidate existing snapshots and force a rebuild.
- `.execute(&executor)` returns `Ok(None)` if no events exist (and no snapshot), `Ok(Some(P))` otherwise. It batches 100 events per call and **errors with `"Too busy"`** if a single load can't finish in one batch — use snapshots for long histories.

### Snapshot trait

Any `P: bitcode::Encode + bitcode::DecodeOwned + ProjectionCursor + Send + Sync` automatically gets a `Snapshot<E>` impl backed by the executor's snapshot table — `Projection::execute` calls `P::restore` before replaying. Override `Snapshot::restore` / `take_snapshot` for custom storage (see the `bank` example which keeps an in-memory `HashMap`).

### `ProjectionAggregator` — emitting events from projections

```rust
impl ProjectionAggregator for AccountView {
    fn aggregator_id(&self) -> String { self.id.to_owned() }
}

// Later:
account.aggregator()?              // builder pre-loaded with id + original_version
    .event(&MoneyDeposited { amount: 100, .. })
    .commit(&executor).await?;
```

`aggregator_version()` reads the version off the projection's cursor — so the version is always in sync with what was replayed.

## Continuous processing — `SubscriptionBuilder`

```rust
use std::time::Duration;
use evento::{Executor, metadata::Event, subscription::{Context, SubscriptionBuilder}};

#[evento::subscription]
async fn notify_on_deposit<E: Executor>(
    ctx: &Context<'_, E>,
    event: Event<MoneyDeposited>,
) -> anyhow::Result<()> {
    // side effects: emails, read-model writes, outbound API calls
    Ok(())
}

let subscription = SubscriptionBuilder::<evento::Sqlite>::new("deposit-notifier")
    .handler(notify_on_deposit())
    .routing_key("accounts")          // only events with this key
    .chunk_size(100)                  // default 300
    .retry(5)                         // default 30, exponential backoff
    .delay(Duration::from_secs(10))   // initial pause before first poll
    .start(&executor)
    .await?;

// graceful shutdown
subscription.shutdown().await?;
```

Key behaviours:

- `.start(&executor)` spawns a tokio task that polls every ~1s, processes events in `chunk_size` batches, and `acknowledge`s cursor after each event.
- `.execute(&executor)` (no spawn) runs once until the cursor reaches the head, then returns.
- `.unretry_start` / `.unretry_execute` — disable retries.
- `.accept_failure()` — log + continue instead of stopping on handler errors.
- `.all()` — process *all* events regardless of routing key (default is `RoutingKey::Value(None)` which only matches events with no routing key).
- `.routing_key(k)` — also prefixes the subscription `key` to `"{routing_key}.{key}"` for per-partition cursor tracking.
- `.skip::<EventType>()` — acknowledge but ignore that event type.
- `.handler(h)` — adding a handler for the *same* event type panics; one handler per (aggregator_type, event_name) per subscription.

Use **`#[evento::subscription_all]`** to receive every event from an aggregate type without decoding the payload (the handler takes `RawEvent<Aggregator>` instead of `Event<EventType>`):

```rust
use evento::metadata::RawEvent;

#[evento::subscription_all]
async fn audit<E: Executor>(ctx: &Context<'_, E>, e: RawEvent<BankAccount>) -> anyhow::Result<()> {
    tracing::info!(name = %e.name, version = e.version, id = %e.aggregator_id, "audit");
    Ok(())
}
```

## Executors

| Type                  | Crate        | When |
|-----------------------|--------------|------|
| `evento::Sqlite`      | `evento-sql` (feature `sqlite`) | `Sql<sqlx::Sqlite>` |
| `evento::Postgres`    | `evento-sql` (feature `postgres`) | `Sql<sqlx::Postgres>` |
| `evento::MySql`       | `evento-sql` (feature `mysql`) | `Sql<sqlx::MySql>` |
| `evento_fjall::Fjall` | `evento-fjall` (feature `fjall`) | Embedded LSM, no external DB |
| `evento::Evento`      | `evento-core` | Type-erased `Arc<Box<dyn Executor>>` |
| `evento::EventoGroup` | feature `group` | Read from multiple executors, write to first |
| `evento::Rw<R, W>`    | feature `rw` | CQRS: separate read/write databases |

Setup pattern (SQL):

```rust
let pool = sqlx::SqlitePool::connect("sqlite:events.db").await?;
let mut conn = pool.acquire().await?;
evento::sql_migrator::new::<sqlx::Sqlite>()?
    .run(&mut *conn, &evento::migrator::Plan::apply_all())
    .await?;
drop(conn);

let executor: evento::Sqlite = pool.into();   // `From<Pool<DB>> for Sql<DB>`
```

`Sql<DB>` is `Clone` (the inner pool is `Arc`), so cheap to pass to handlers, web state, etc.

Fjall:

```rust
let executor = evento_fjall::Fjall::open("./events")?;       // creates dirs if missing
// or: Fjall::from_keyspace(fjall::Config::new(path).open()?) for custom config
executor.persist()?;                                          // force fsync
```

## Metadata

`Metadata` is a bitcode-serialized `HashMap<String, Vec<u8>>` keyed by string, plus a ULID `id`.

```rust
use evento::metadata::Metadata;

let mut meta = Metadata::default();
meta.set_requested_by("user-123");
meta.set_requested_as("impersonated-789");

create()
    .event(&MyEvent { .. })
    .metadata_from(&meta)
    .commit(&executor).await?;

// On the read side, `Event<MyData>` derefs to `crate::Event`, so:
if let Ok(by) = event.metadata.requested_by() { ... }
let custom: MyType = event.metadata.try_get("KEY")?;
```

`MetadataError::NotFound | Decode(bitcode::Error)`.

## Cursor pagination

All reads through the `Executor::read` API use GraphQL-style cursors.

```rust
use evento::{cursor::Args, ReadAggregator};

let page = executor.read(
    Some(vec![ReadAggregator::id("crate/BankAccount", &account_id)]),
    None,                                  // RoutingKey filter (None = ignore)
    Args::forward(20, None),               // first 20
).await?;

let next = executor.read(
    Some(vec![ReadAggregator::aggregator("crate/BankAccount")]),
    None,
    Args::forward(20, page.page_info.end_cursor.clone()),
).await?;
```

`ReadAggregator` constructors:

- `::new(type, id, event_name)` — exact match
- `::aggregator(type)` — all events of a type
- `::id(type, id)` — all events of one instance
- `::event(type, event_name)` — all events of one name

`Args::forward(n, after)` / `Args::backward(n, before)`; `.is_backward()`, `.limit(cap)`.

`ReadResult<T>` has `edges: Vec<Edge<T>> { cursor, node }` and `page_info: PageInfo { has_previous_page, has_next_page, start_cursor, end_cursor }`.

For custom paginated read-models on top of SQL, use `evento::sql::Reader` (it derefs to a sea-query `SelectStatement`) plus `#[derive(evento::Cursor)]` on your row struct — see [`executors.md`](./executors.md).

## Common pitfalls

- **`Projection::new::<MyEvents>` vs `Projection::new::<MyEvents>(id)`** — pass the aggregator *enum* name as the type parameter (it became a unit struct), not the event struct.
- **Forgetting `original_version`** on appends — it defaults to `0`, which is correct only for the very first event of a new aggregate. Pass the projection's current version (or use `ProjectionAggregator::aggregator()` which fills it in).
- **Aggregator type strings are `"{cargo_pkg_name}/{Enum}"`** — embed-renaming the crate breaks reads of old events. Use a stable name for the events crate.
- **Routing key is locked after the first event**. Setting `.routing_key("x")` on an aggregate that already has events is a no-op.
- **Default `Metadata::default()`** generates a new ULID *every call* — clone or build once if you want a stable id across a batch.
- **`Projection::execute` reads at most 100 events at a time**; on long histories without snapshots you'll get `Err("Too busy")`. Take a snapshot or use a subscription.
- **`SubscriptionBuilder::handler` panics on duplicate handlers** for the same (aggregator_type, event_name). One handler per event type per subscription.
- **`safety_check` is off by default** on both `Projection` and `SubscriptionBuilder` — events with no handler are silently skipped. Turn it on to catch typos.
- **`Default` for the projection state must be sensible** — `Projection::execute` builds the initial state via `P::default()` if no snapshot exists. Required fields with no defaults will panic during decode/replay.
- **Handler signature is strict.** Projection handlers take `(event: Event<E>, projection: &mut P) -> anyhow::Result<()>`; subscription handlers take `(ctx: &Context<'_, E>, event: Event<EV>) -> anyhow::Result<()>` and must be generic over `E: Executor`. Deviating produces opaque macro errors — see [`macros.md`](./macros.md).
- **`#[evento::subscription_all]` events name is `"all"`** internally; the dispatch key `{aggregator_type}_all` shadows per-event handlers, so don't mix `subscription_all` with per-event `subscription` handlers on the same subscription unless that's the intent.
